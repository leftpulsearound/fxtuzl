# TechRef Navigator

TechRef Navigator 是一个面向开发者、技术研究人员与开源项目维护者的结构化技术资源导航工具。本项目并非一个传统意义上的应用程序或框架，而是一个精心编排的外部技术文章与参考资料索引系统，旨在解决开发者在学习与调试过程中面临的“信息过载”与“优质内容分散”问题。

本项目定位为技术决策支持与知识加速器。通过对海量外部技术文档（涵盖前后端工程、数据库调优、DevOps 实践、编程语言底层原理等领域）进行深度链接与分类挂载，用户可以在统一的入口处快速检索到特定问题的解决方案或特定技术栈的深度解析。本项目特别适合处于技术选型阶段的技术负责人、需要快速排查线上问题的运维工程师，以及希望系统化构建技术图谱的进阶学习者。当前资源批次为第 3/56 批，累计收录并索引外部技术参考链接达 180 条。

## 功能概览

**URL 严格原样收录**：系统底层基于原始 URL 字符串进行存储与展示，禁止对协议头（http/https）、主机名（含 www 子域）及路径大小写进行任何形式的规范化改写，确保链接指向的确定性。

**多维度分类索引**：内置启发式分类引擎，可根据 URL 路径特征（如数字 ID 分段、文件扩展名大小写）自动将链接归入后端实现、前端交互、运维监控、算法理论等不同技术板块。

**纯静态资源映射**：项目核心为 Markdown 格式的链接资产清单，无需数据库与后端运行时依赖，可通过任意 HTTP 服务器直接托管，符合文档即代码的理念。

**离线阅读友好**：所有链接资源以清单形式固化在仓库中，用户可通过 `git clone` 全量拉取至本地，配合个人知识管理工具进行离线批注与二次整理。

**变更追踪机制**：资源列表随项目迭代持续更新，每次新增或删除链接均通过 Git 提交记录保留审计痕迹，便于团队协作审查外部引用源的可靠性。

**快速模糊检索**：用户可利用编辑器或 `grep` 命令行工具对 URL 中的数字 ID、文件名后缀（如 `.sHtML`）进行快速文本匹配，定位相关文章。

**扩展性数据结构**：资源列表采用纯文本行格式存储，支持用户通过脚本自动化提取 URL 进行可用性检测或批量元数据抓取。

## 应用场景

**技术预研与方案评估**：当团队计划引入新的中间件或框架时，架构师可通过本项目快速查阅相关技术博客的实践总结，了解社区中该技术的常见坑点与性能边界，辅助技术选型决策。

**线上故障紧急排查**：运维人员在收到服务告警后，可利用本项目的检索功能快速定位涉及特定错误码（如 502、Connection Timeout）或特定数据库异常（如死锁、索引失效）的外部讨论贴，借鉴他人处理经验缩短故障恢复时间。

**新人入职培训知识储备**：项目维护者可引导新入职的开发人员将本项目作为技术阅读清单，通过系统浏览索引下的各类深度技术文章，加速其对团队所用技术栈（从底层网络协议到上层业务框架）的整体认知。

**开源项目文档补充**：开源项目作者可在 README 或 Wiki 中引用本项目中的特定链接，作为外部参考资料的延伸阅读部分，减轻自身撰写长篇技术背景文档的负担。

## 快速开始

以下步骤指导您在本地环境快速部署并运行 TechRef Navigator 静态站点。

```bash
# 克隆项目仓库至本地
git clone https://github.com/techref-navigator/techref-navigator.git

# 进入项目根目录
cd techref-navigator

# 安装依赖（此处假定使用 Node.js 生态的 http-server 作为静态服务器，亦可用任何其他静态托管方案）
npm install -g http-server

# 启动静态服务，默认监听 8080 端口
http-server ./public -p 8080
```

执行完毕后，在浏览器中访问 `http://127.0.0.1:8080` 即可查看资源索引首页。所有收录的 URL 链接均位于 `public/resources/` 目录下的 Markdown 文件中。

## 安装要求

| 依赖 | 必需 | 说明 |
| :--- | :--- | :--- |
| Git | 是 | 用于克隆仓库及版本回溯，建议版本 2.20.0 以上 |
| Node.js | 否 | 仅当使用项目内置的静态服务脚本时需要，版本要求 >= 12.0.0 |
| http-server | 否 | 轻量级本地静态文件服务，非必须，可用 Nginx/Apache 替代 |
| 现代 Web 浏览器 | 是 | 用于渲染 Markdown 索引页，推荐 Chrome/Edge/Firefox 最新版 |
| 磁盘空间 | 是 | 项目纯文本存储，占用空间小于 10 MB，无额外存储压力 |
| 网络连接 | 否 | 项目本身无需联网运行，但所索引的外部 URL 需联网访问 |
| make | 否 | 若需运行项目自带的链接格式校验脚本，建议安装 GNU Make |
| Shell 环境 | 否 | 运行辅助脚本（如 URL 提取统计）建议使用 Bash 5.0 以上 |

│   ├── index.html                   # 项目入口页面，包含资源列表的渲染容器
│   ├── css/                         # 样式表目录，存放基础布局与响应式设计
│   │   └── style.css                # 主样式文件，定义卡片、表格与代码块样式
│   ├── js/                          # 前端交互脚本目录

## 贡献指南

我们欢迎社区成员提交新的高质量技术参考链接或对本项目的索引分类提出改进建议。为确保资源库的长期可用性与内容质量，请遵循以下步骤。

**第一步：阅读设计原则与分类规范**

在提交任何内容之前，请务必浏览 `docs/design-principles.md` 与 `docs/classification-rules.md`。理解本项目禁止对 URL 做任何形式改写的底层原因，以及数字 ID 与文章主题之间的映射关系，这有助于您准确判定新链接应归属的批次与分类。

**第二步：Fork 仓库并创建特性分支**

通过 GitHub 的 Fork 功能将本项目复制至您的个人账户下，随后克隆至本地，并基于最新的 main 分支创建一个新的特性分支，分支命名建议采用 `feature/add-batch-3-links` 或 `fix/url-typo` 的格式。

**第三步：编辑资源文件并遵守格式约束**

在 `public/resources/` 目录下找到对应的批次文件（如当前批次为 batch-3.md），在文件末尾追加新的链接。每行仅允许放置一条 URL，且必须严格复制原始来源的地址，严禁修改协议头、域名大小写及路径后缀。提交前请运行 `scripts/validate-urls.sh` 脚本进行本地格式检查。

**第四步：编写清晰的提交信息与拉取请求**

使用 Git 提交您的更改，提交信息应简明扼要说明本次新增或修改的链接数量及主题分类。随后将特性分支推送至您的远程仓库，并通过 GitHub 界面发起 Pull Request 至本仓库的 main 分支。请在 PR 描述中逐一列出新增链接的原始出处及推荐理由。

**第五步：等待代码审查与合并**

项目维护者将定期审查 PR，检查链接是否有效、格式是否符合规范以及内容是否与现有索引重复。审查通过后，您的提交将被合并至主分支，并自动触发 CI 流水线重新构建静态站点。

## 常见问题

**问：为什么项目要求 URL 必须原样输出，不允许补全协议头或规范化大小写？**

答：这是本项目核心的设计契约。部分外部技术博客或文档站点的服务器对 URL 的大小写敏感（尤其是在 Linux/Apache 环境下），且 `http` 与 `https` 可能指向不同版本的内容或存在重定向策略。强制原样输出旨在确保用户复制链接后，在浏览器中访问到的内容与贡献者当时验证的页面完全一致，避免因自动改写导致的 404 或内容错位问题。我们的 CI 流水线会严格拦截任何试图修改 URL 格式的提交。

**问：如何快速判断某个链接是否已经被收录？**

答：您可以使用项目根目录下的 `scripts/count-links.sh` 脚本配合 `grep` 命令进行快速检索。例如，要查找包含特定数字 ID 的链接，可执行 `grep -r "特定数字ID" public/resources/`。此外，项目维护团队正在规划基于 JSON 索引的全文检索功能，预计在后续批次中上线。

**问：我发现某个已收录的链接已经失效（返回 404 或连接超时），应该如何处理？**

答：我们高度依赖社区维护来保持资源列表的有效性。若您发现死链，请通过 GitHub Issues 提交报告，并在标题中注明 `[Dead Link]` 及对应的 URL。维护人员核实后会将其从资源文件中移除，并在 CHANGELOG 中记录清理操作。我们也鼓励贡献者提交替代性的有效链接来填补内容空缺。
