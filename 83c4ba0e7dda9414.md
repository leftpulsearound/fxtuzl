# ResourceHub

ResourceHub 是一个面向开发者、技术研究人员与内容创作者的轻量级技术资源导航与外链聚合平台。该项目定位于对分散在网络各处的优质技术文章、教程、案例分析与开发文档进行系统性归集与索引，帮助用户以最低的检索成本获取高价值的技术参考资料。

ResourceHub 不生产内容，而是通过严格的链接遴选与分类机制，将来自同一信源的高质量技术文章按主题、难度与适用场景进行组织。项目特别适用于以下人群：需要快速查阅特定技术问题解决方案的开发者、希望跟踪技术文章更新动态的研究者、以及需要为团队建立内部技术知识库的架构师。通过 ResourceHub，用户可以将原本零散的书签、收藏夹与笔记整合为结构清晰、可复用、可共享的外部资源索引体系。

## 功能概览

**链接分类索引** 支持按技术领域、文章类型、发布批次对收录的 URL 进行多维度归类，用户可通过目录快速定位到特定主题的资源集合。

**资源元信息提取** 自动从 URL 结构中解析文章标识符与类型后缀，辅助用户在不打开页面的情况下初步判断资源内容属性。

**批量导入与导出** 支持将外部链接列表批量导入系统，并能够将索引数据导出为多种格式（JSON、CSV、Markdown），便于与其他工具链集成。

**全文检索过滤** 基于 URL 关键字与文章编号提供快速的搜索过滤能力，帮助用户在数百条链接中精准定位目标资源。

**状态标记与阅读进度跟踪** 允许用户对已查阅、待查阅、收藏或过期的链接进行自定义标记，并记录每次访问的时间戳。

**多用户协作支持** 提供基于角色的访问控制，团队成员可共享同一资源池并分别维护各自的阅读状态与备注信息。

**数据快照与回滚** 每次批量更新操作前自动创建索引快照，支持一键回滚至任意历史版本，避免误操作导致的数据丢失。

**RESTful API 接口** 提供完整的 HTTP API，允许第三方工具或脚本程序以编程方式查询、新增或删除资源链接，实现自动化运维。

## 应用场景

技术团队内部知识库建设。开发团队可将日常遇到的有价值的外部技术文章、问题排查记录与架构设计参考统一收录至 ResourceHub，新成员入职时可直接通过该平台了解团队所依赖的技术资源生态，显著降低知识传递成本。

个人开发者学习路径管理。独立开发者或自学者可将分散在浏览器收藏夹、笔记软件和即时通讯群组中的技术链接集中迁移至 ResourceHub，按照学习阶段（入门、进阶、实战）或技术栈（前端、后端、运维、AI）进行分类，形成可追溯的个人学习资料库。

技术社区内容聚合与推荐。技术博客作者或社区运营人员可利用 ResourceHub 对某一技术专题下的外部优质内容进行整理与推荐，以结构化列表的形式发布给读者，替代传统的零散推荐方式，提升阅读体验与内容复用率。

## 快速开始

以下指令演示如何从代码仓库克隆 ResourceHub、安装依赖并启动开发服务。

```bash
git clone https://github.com/resourcehub/resourcehub.git
cd resourcehub
npm install
npm run dev
```

执行上述命令后，开发服务器默认启动于 http://localhost:3000。访问该地址即可进入 ResourceHub 的本地实例，开始导入或浏览资源链接。生产环境部署请参考 `docs/deployment.md` 中的说明。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 运行时环境，建议使用 nvm 管理版本 |
| npm | 9.x 或以上 | 包管理器，用于安装项目依赖 |
| SQLite | 3.35 或以上 | 默认嵌入式数据库，无需额外安装 |
| Redis | 7.0 或以上 | 可选依赖，用于会话存储与缓存加速 |
| Nginx | 1.22 或以上 | 生产环境反向代理与静态资源服务推荐 |
| PM2 | 5.0 或以上 | 生产环境进程守护与自动重启工具 |

## 项目结构

```
resourcehub/
├── src/
│   ├── api/                     # RESTful API 路由与控制器定义
│   │   ├── routes/              # 按资源类型划分的路由模块（links, users, tags）
│   │   └── middleware/          # 身份验证、日志记录与限流中间件
│   ├── core/                    # 核心业务逻辑层
│   │   ├── importers/           # 批量链接导入器（支持 CSV / JSON / Markdown）
│   │   ├── exporters/           # 数据导出器（JSON / CSV / Markdown）
│   │   └── snapshot/            # 快照创建与回滚管理模块
│   ├── models/                  # 数据模型定义（Link, User, Tag, Snapshot）
│   │   ├── sqlite/              # SQLite 数据库适配器与迁移脚本
│   │   └── redis/               # Redis 缓存操作封装
│   ├── services/                # 外部服务集成（全文检索、邮件通知）
│   │   ├── search/              # 基于 Lunr.js 的本地全文检索引擎
│   │   └── notifier/            # 状态变更通知服务（邮件 / Webhook）
│   ├── ui/                      # 前端界面组件（React + Tailwind）
│   │   ├── pages/               # 主页面：列表视图、详情视图、管理后台
│   │   ├── components/          # 可复用 UI 组件（LinkCard, FilterBar, Pagination）
│   │   └── hooks/               # 自定义 React Hooks（useLinks, useAuth）
│   └── utils/                   # 通用工具函数（URL 解析、日期格式化、校验器）
├── config/                      # 环境配置文件（development, staging, production）
│   ├── default.json             # 默认配置项（端口、数据库路径、缓存TTL）
│   └── custom-environment-variables.json  # 环境变量映射表
├── tests/                       # 单元测试与集成测试用例
│   ├── unit/                    # 模型与工具函数测试
│   └── integration/             # API 端点与数据库交互测试
├── docs/                        # 完整项目文档（快速开始、API参考、运维手册）
├── scripts/                     # 运维脚本（数据备份、索引重建、批量迁移）
├── public/                      # 静态资源（favicon, robots.txt, 错误页面）
├── package.json                 # npm 项目清单与脚本定义
├── .env.example                 # 环境变量示例文件
└── README.md                    # 项目说明文档（本文件）
```

## 贡献指南

1. 在 GitHub 上 Fork 本仓库，并克隆到本地开发环境。确保使用 Node.js 18.x 版本，执行 `npm install` 安装全部依赖。

2. 新建功能分支，分支命名采用 `feature/功能简述` 或 `fix/问题简述` 格式。所有代码变更需通过 ESLint 与 Prettier 检查，并确保现有单元测试全部通过。

3. 若新增外部资源链接，请通过项目提供的导入工具或 API 接口进行操作，避免直接编辑数据库文件。对于新增功能模块，需同步补充对应的单元测试与文档说明。

4. 提交代码时使用语义化提交信息（如 `feat: 增加按标签过滤链接功能`、`fix: 修复快照回滚时索引丢失问题`）。提交前执行 `npm run lint` 与 `npm run test` 进行自检。

5. 发起 Pull Request 至主仓库的 `main` 分支，并在 PR 描述中清晰说明变更目的、影响范围与测试结果。PR 合并前需至少一名核心维护者审核通过。

## 常见问题

问：项目是否支持 PostgreSQL 或 MySQL 作为生产数据库？

答：当前正式版本仅内置 SQLite 适配器，因其零配置特性适合快速部署与中小规模数据量场景。若需使用 PostgreSQL，可基于 `src/models/` 中的抽象数据访问层自行扩展，项目已预留数据库适配器接口，社区贡献的 PostgreSQL 实现正在规划中。

问：如何批量更新已导入链接的元信息（如分类标签或状态标记）？

答：可通过管理后台的批量编辑功能，选中多个链接后统一修改标签或状态。对于高级操作，建议使用 REST API 的批量更新端点 `PATCH /api/links/batch`，请求体传入需要更新的链接 ID 列表与字段键值对，系统会以事务方式执行更新，部分失败时返回详细的错误报告。

问：快照功能会占用大量磁盘空间吗？

答：快照仅存储索引的元数据结构（链接列表与用户标记），不包含外部文章的实际内容，因此每个快照的大小通常在几百 KB 至数 MB 之间。系统默认保留最近 30 个快照，超过数量上限时自动轮转删除最旧快照。若需调整保留策略，可修改 `config/default.json` 中的 `snapshot.maxCount` 参数。
