# WebArchive Navigator

WebArchive Navigator 是一个面向技术研究、历史数据回溯与互联网内容分析的专业外链资源聚合平台。该项目系统性地收集并整理了互联网上分散于各类技术博客、新闻站点与个人网站中的历史文章链接，致力于为研究人员、开发者以及数据分析师提供一套结构清晰、可追溯、可持续维护的 URL 索引体系。

本项目定位于技术资源的中转枢纽，而非内容存储仓库。它不直接托管或缓存任何第三方页面内容，而是通过精细化的链接分类与元数据标注，帮助用户在海量网络信息中快速定位到特定主题、特定时间段或特定来源的原始文章。项目内置了完整的资源导航框架与文档化结构，支持用户基于本地环境快速部署一份可检索、可扩展的外链目录服务。

## 功能概览

**历史链接索引** 项目收录了超过 180 条源自 wap.blog.kmvdvi.cn 域名的深度文章链接，每条链接均保留原始 URL 结构与参数，确保指向内容的可解析性。

**分类化资源目录** 所有外链按照文章主题、发布时间或内容类型进行逻辑分组，便于用户在特定技术领域内进行集中浏览与筛选。

**本地化快速部署** 提供完整的克隆、安装与运行脚本，用户可在数分钟内于本地环境启动一份私有的资源导航站点，无需依赖外部数据库或云服务。

**可扩展的数据模型** 项目核心数据基于结构化文本文件存储，用户可自行增删 URL 条目或调整分类标签，所有修改均通过标准配置文件完成。

**自动化链接状态检测** 内置基础的链接可达性检查工具，支持定时或手动触发，帮助维护人员及时发现失效链接并作出标注。

**纯静态页面渲染** 前端采用零依赖的 HTML 与 CSS 渲染方案，无需 JavaScript 框架支持，确保页面在各类浏览器与低带宽环境下均能快速加载。

**完整的文档体系** 项目附带详尽的安装要求、目录结构说明、贡献指南与常见问题解答，降低新用户的上手门槛，同时为维护者提供清晰的协作规范。

## 应用场景

技术文献回溯研究
研究人员可通过本项目提供的索引条目，快速定位到特定技术主题下的历史文章。例如，当需要追踪某开源框架早期版本的讨论或某算法在移动端的首次实践记录时，可直接依据分类导航跳转至对应外链，无需在搜索引擎中反复调整关键词。

开发团队的内部知识库补充
开发团队可将本项目部署为内部文档站点的外链补充模块。当团队 Wiki 或 Confluence 中缺少某类外部参考资源时，可直接将本项目的分类链接页嵌入 iframe 或通过菜单项引用，使成员在查阅内部文档的同时能便捷地获取外部上下文信息。

数据分析与互联网内容演变观察
数据分析师可利用项目收录的 URL 列表作为样本集，分析特定域名下文章 ID 的分配规律、发布时间密度或路径结构特征。通过对 180 余条样本的批量访问与响应记录统计，可初步评估该站点的内容更新频率与服务器稳定性。

个人知识管理系统的外链模块
个人笔记工具（如 Obsidian、Notion 或 Logseq）的使用者可将本项目导出的 URL 清单作为外部书签源导入，配合标签系统与全文检索功能，构建一套兼顾本地笔记与网络参考资料的个人知识网络。

## 快速开始

以下命令序列适用于 Linux、macOS 及 Windows WSL 环境。请确保系统已安装 Git 与 Python 3.8 或更高版本。

```bash
# 克隆仓库至本地
git clone https://github.com/example/webarchive-navigator.git
cd webarchive-navigator

# 安装依赖（使用 pip 与虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 运行本地导航服务（默认监听 8000 端口）
python server.py
```

执行完毕后，在浏览器中访问 http://127.0.0.1:8000 即可进入资源导航主页。如需修改端口，可编辑 server.py 中的 `PORT` 变量或通过命令行参数指定。

## 贡献指南

提交代码或数据更新前，请仔细阅读以下步骤，确保变更符合项目规范。

1.  Fork 本仓库至个人账户，并在本地克隆 fork 后的副本。新建一个以功能或修复命名的分支，例如 `feat/add-category-tutorial` 或 `fix/link-checker-timeout`，避免在主分支上直接修改。

2.  对 `data/links.csv` 或 `config.yaml` 进行增删改时，务必保持 CSV 的列顺序与 YAML 的缩进一致。新增链接建议先通过 `scripts/check_links.py` 验证可达性，确认 HTTP 状态码为 200 后再提交。

3.  提交前需运行全量单元测试，确保 `tests/` 目录下的测试用例全部通过。若新增了功能模块，请同步编写对应的测试脚本，覆盖至少 80% 的新增代码路径。

4.  Commit message 应遵循约定式提交格式，使用 `feat:`、`fix:`、`docs:`、`chore:` 等前缀，并附带简明扼要的变更描述。严禁提交包含敏感信息（如内网 IP、密码、令牌）的文件。

5.  向本仓库的 `main` 分支发起 Pull Request，并在 PR 描述中注明变更动机、影响范围以及是否涉及破坏性改动。至少需要一位项目维护者审核通过后，方可合并。

## 常见问题

**Q：为什么项目不直接缓存文章内容，而只提供外链？**

A：本项目的核心定位是导航索引，而非内容存储。直接缓存第三方文章内容可能引发版权争议，且会显著增加项目的存储成本与维护复杂度。通过外链方式，用户始终能够访问到源站的最新版本内容，同时也避免了因缓存陈旧信息而导致的误导。

**Q：本地运行导航服务时，页面上的链接无法打开，应如何排查？**

A：首先确认运行 `server.py` 的终端窗口没有报错，且 8000 端口未被其他进程占用。其次，检查 `data/links.csv` 中的 URL 是否包含换行符或特殊空格字符。如果链接本身已失效，可在 `config.yaml` 中将 `link_check_enabled` 设为 `true`，然后运行 `scripts/check_links.py` 生成状态报告，失效链接会以红色标记在页面上。

**Q：如何批量导入新的外链资源？**

A：准备一份符合 `data/links.csv` 列定义（URL、category、tags、source、added_date）的 CSV 文件，然后执行 `scripts/import_from_csv.py --input your_file.csv --merge`。该脚本支持去重合并，若新链接与现有链接的 URL 完全一致，则不会重复添加，仅更新其分类标签与时间戳。
