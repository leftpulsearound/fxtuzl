# H5 Blog Archive Bridge

H5 Blog Archive Bridge 是一个面向技术内容聚合与历史数据归档的开源工具集，专注于对 h5.blog.kmvdvi.cn 平台下的文章详情页进行结构化索引与本地镜像管理。该项目定位为技术资源与外链汇总中间件，服务于需要批量抓取、离线查阅、全文检索或二次加工该平台公开文章内容的研究者、开发者和数据分析人员。

该项目解决的核心问题在于：原始平台的文章详情页以静态 HTML 形式发布，缺乏统一的目录索引机制，且 URL 规则中包含大小写混合的 .sHtML 后缀，给自动化采集与归档带来不便。H5 Blog Archive Bridge 提供了一套可复用的解析模板、链接管理脚本和元数据抽取方案，帮助用户将分散的文章链接转化为可查询、可追溯、可持久化存储的本地知识库。

## 功能概览

**批量链接归一化处理**：自动识别并解析 h5.blog.kmvdvi.cn 域名下的文章详情页 URL 格式，支持对 .sHtML 后缀的大小写不敏感匹配，输出标准化访问路径。

**文章元数据抽取**：从 HTML 文档中提取标题、发布时间、正文摘要、分类标签等关键字段，生成 JSON 或 CSV 格式的结构化数据清单。

**增量更新与去重机制**：基于本地缓存记录已抓取的文章 ID，避免重复请求，支持定时任务触发的增量同步模式。

**多格式导出支持**：提供 Markdown、HTML 离线包、JSON Lines 三种导出格式，满足文档阅读、静态站点生成和数据导入数据库等不同使用需求。

**内置本地预览服务**：集成轻量级 HTTP 服务器，可在本地启动预览界面，按时间或分类浏览已归档的文章列表。

**链接可用性检查**：周期性巡检已收录 URL 的响应状态码，生成失效链接报告，便于维护者及时更新资源库。

## 应用场景

**技术博客本地镜像**：开发者可将该平台上的系列技术文章批量下载到本地，在无网络环境中通过内置预览服务进行全文检索和阅读，适合出差、离线办公或网络受限环境。

**数据分析与文本挖掘**：研究人员可使用本项目的元数据抽取功能，将文章标题、正文和标签导出为结构化数据集，用于词频统计、主题建模或趋势分析等自然语言处理任务。

**内容聚合站外链备份**：运维人员可将项目作为外链汇总站的核心组件，定期抓取文章链接并生成索引页面，避免因原始平台内容调整或链接变动导致的外链失效问题。

**知识库初始化种子**：团队内部知识管理系统的初始化阶段，可通过本项目批量导入历史文章作为知识库的种子数据，再结合人工审核补充内部文档。

## 快速开始

以下命令演示如何从 GitHub 克隆项目、安装依赖并运行首次全量抓取任务。

```bash
git clone https://github.com/your-org/h5-blog-archive-bridge.git
cd h5-blog-archive-bridge
pip install -r requirements.txt
python main.py --mode fetch --input urls.txt --output ./data
```

其中 urls.txt 为存放文章链接列表的文本文件，每行一个 URL。首次运行时会自动创建 data 目录，并生成 metadata.json 作为主索引文件。

## 贡献指南

1. 查阅 issue 列表，认领未被指派的待办任务，或提交新的 bug 报告与功能建议。新提交的 issue 需附带可复现的步骤或明确的使用场景描述。

2. Fork 本仓库，在 dev 分支上创建功能分支，分支命名遵循 feature/功能简述 或 fix/问题编号 的格式。提交前运行 pytest 确保所有已有测试用例通过，并为新增代码编写对应的单元测试。

3. 代码风格遵循 PEP 8 规范，使用 black 工具进行格式化。提交信息采用约定式提交格式，类型包括 feat、fix、docs、style、refactor、test、chore，正文说明变更原因和影响范围。

4. 完成开发后提交 Pull Request 到 dev 分支，PR 描述中需关联对应的 issue 编号，并附上手动测试的结果截图或日志片段。核心模块的变更需由至少一位维护者审核后合并。

5. 文档类贡献（修正错别字、补充示例、完善 API 说明）可直接提交 PR 到 main 分支，无需经过完整的功能性测试流程，但需确保 MkDocs 构建无警告。

## 常见问题

**问：抓取过程中遇到 HTTP 403 或 429 状态码，应如何处理？**

答：该项目已内置指数退避重试策略，默认最大重试次数为 3 次。若持续返回 429，请在配置文件中调整请求间隔参数（request_interval），建议设置为 2 至 5 秒。若返回 403，需检查请求头中的 User-Agent 是否被目标服务器屏蔽，可通过 middleware 模块切换预设的 User-Agent 池。

**问：导出的 JSON Lines 文件体积过大，如何分割或压缩？**

答：writer 模块支持按条数或按文件大小自动切割，可在配置文件中设置 max_records_per_file 或 max_size_mb 参数。切割后的文件会以 .part 序号后缀区分。若需压缩存储，可将输出目录挂载至支持透明压缩的文件系统，或使用 scripts 目录下的 gzip_rotate.sh 脚本进行事后压缩。

**问：如何仅抓取指定日期范围的文章？**

答：该项目目前不支持基于时间范围的远程过滤，因为原始平台未提供按时间查询的接口。建议先运行全量抓取，再通过 parser 模块抽取的 published_date 字段在本地进行过滤。后续版本计划增加增量模式，仅抓取上次运行后新发布的文章，可通过检查本地索引中已存在的最大文章 ID 来实现。
